常用设计模式

设计模式在 C++ 面试里通常不是为了考你背名字，而是看你能否把“为什么这样设计”“替代方案是什么”“代价在哪里”讲清楚。 真正高质量的回答，不是只说定义，而是能结合场景说明：什么问题适合用模式，什么问题不该为了模式而模式。

为什么面试官喜欢问设计模式

因为设计模式本质上是在考抽象能力，而不是考记忆力。面试官想知道的是：当需求变化、分支增多、模块耦合变重时，你是否能把变化点拆出来， 让代码更容易扩展、测试和维护。

考抽象

你能否从具体业务中抽出稳定点和变化点，而不是把所有逻辑堆进一个类。

考取舍

是否知道模式也有成本，简单问题没必要上复杂抽象。

考工程感

能否说清楚模式与 C++ 多态、组合、所有权管理之间的关系。

一个很稳的开场方式是：设计模式不是固定套路，而是对常见设计问题的经验总结。面试里更重要的是说明“为什么用”“何时不用”“用了以后好在哪，代价又是什么”。

单例模式

单例模式的目标是保证一个类在系统中只有一个实例，并提供一个全局访问点。它很常见，但也是最容易被追问“你真的会用吗”的模式之一。

维度 说明
详细定义 确保某个类只有一个实例，并向整个系统提供访问该实例的统一入口。
常见用途 日志器、配置中心、线程池管理器、全局资源注册表。
面试高频追问 线程安全怎么保证、懒汉式和饿汉式区别、为什么很多时候不推荐滥用单例。
客户端不会自己创建多个对象，而是都拿到同一个实例。
class Logger {
public:
    static Logger& instance() {
        static Logger logger;
        return logger;
    }

    void log(const std::string& msg) {
        std::cout << "[LOG] " << msg << "\n";
    }

private:
    Logger() = default;
    Logger(const Logger&) = delete;
    Logger& operator=(const Logger&) = delete;
};
在 C++11 及之后，函数内局部静态变量的初始化是线程安全的，所以这种写法是实现单例的常见推荐方式。
单例的最大问题不是“写不出来”，而是容易把全局状态藏起来，导致测试困难、依赖不透明、模块耦合变强。所以它通常适合真正唯一且全局共享的资源，不适合所有“图省事”的场景。
面试回答模板：单例模式用于保证某个类在系统中只有一个实例，并提供全局访问点。C++ 中常见实现是函数内静态对象。它适合日志、配置等确实全局唯一的资源，但要警惕隐藏依赖和测试困难的问题。

工厂模式

工厂模式的核心是把“对象如何创建”从“对象如何使用”里分离出来。它在 C++ 里尤其常见，因为对象创建经常伴随多态、资源所有权和初始化参数的管理。

简单工厂

由一个工厂类根据参数决定创建哪种对象。好理解、好上手，但工厂内部容易出现新的 if-else。

工厂方法

把创建职责延迟到子类工厂中。扩展性更好，更符合开闭原则，但结构会更复杂。

客户端不直接依赖具体产品构造过程，而是通过工厂拿到抽象产品。
class Shape {
public:
    virtual ~Shape() = default;
    virtual void draw() const = 0;
};

class Circle : public Shape {
public:
    void draw() const override {
        std::cout << "draw circle\n";
    }
};

class Rectangle : public Shape {
public:
    void draw() const override {
        std::cout << "draw rectangle\n";
    }
};

class ShapeFactory {
public:
    static std::unique_ptr<Shape> create(const std::string& type) {
        if (type == "circle") {
            return std::make_unique<Circle>();
        }
        if (type == "rectangle") {
            return std::make_unique<Rectangle>();
        }
        return nullptr;
    }
};
在 C++ 里，工厂经常和 std::unique_ptr 一起出现，因为创建职责往往伴随着所有权交付。谁创建、谁返回、谁接管生命周期，这些点面试官很爱继续追问。
典型实例：插件系统根据配置创建不同解析器；GUI 框架根据类型创建按钮、文本框；游戏里根据敌人种类创建不同角色对象。

策略模式

策略模式定义一组可以互换的算法，并把每个算法独立封装起来，让它们在运行时可以替换。它最适合“目标一致、算法不同”的场景。

本质

封装算法族，让上下文依赖抽象而不是具体算法。

典型场景

折扣计算、支付方式、路径规划、角色 AI、排序规则切换。

面试关键词

运行时替换、组合优于继承、消灭复杂分支、开闭原则。

Context 只负责调用策略，不关心策略内部细节。
class PricingStrategy {
public:
    virtual ~PricingStrategy() = default;
    virtual double calc(double price) const = 0;
};

class VipStrategy : public PricingStrategy {
public:
    double calc(double price) const override {
        return price * 0.8;
    }
};

class CashBackStrategy : public PricingStrategy {
public:
    double calc(double price) const override {
        return price >= 200.0 ? price - 30.0 : price;
    }
};
这类题目面试里最好顺手补一句：策略模式不只是“用了多态”，而是把变化的行为从上下文中独立出去，让业务主流程保持稳定。
相关实例：商城结算切换折扣算法；地图导航切换最短距离、最少时间、最低收费；游戏敌人切换激进、防守、逃跑行为。

观察者模式

观察者模式用于建立对象之间的一对多依赖关系。当主题对象状态变化时，会自动通知所有依赖它的观察者。它特别适合事件驱动场景。

角色 职责
Subject 维护观察者列表，状态变化时发通知。
Observer 定义更新接口，接收通知并执行响应逻辑。
ConcreteObserver 实现具体更新动作，如刷新 UI、写日志、发告警。
主题对象状态变化后，统一通知多个观察者，各自处理自己的后续逻辑。
class Observer {
public:
    virtual ~Observer() = default;
    virtual void update(int value) = 0;
};

class Subject {
public:
    void attach(Observer* observer) {
        observers_.push_back(observer);
    }

    void set_value(int value) {
        value_ = value;
        notify();
    }

private:
    void notify() {
        for (auto* observer : observers_) {
            observer->update(value_);
        }
    }

    int value_ = 0;
    std::vector<Observer*> observers_;
};
观察者模式在 C++ 里要格外注意生命周期。如果 Subject 持有裸指针，而某个 Observer 提前销毁，就可能出现悬空指针。实际工程里常配合弱引用、事件总线或更明确的订阅取消机制。
相关实例：GUI 按钮点击事件、配置变更通知、股票价格推送、消息订阅系统、游戏任务系统的状态广播。

适配器模式

适配器模式用于把一个类的接口转换成客户端希望的另一个接口。它特别适合“旧系统接口不能改，但新系统又想统一调用方式”的场景。

核心问题

旧接口能工作，但和当前代码约定不兼容，直接使用会让系统到处出现特殊分支。

解决方式

增加一个中间适配层，把旧接口包装成新接口，而不是去修改原有被适配对象。

适配器本身不改变原功能，只负责把接口形状对齐。
class OldPrinter {
public:
    void old_print(const std::string& text) {
        std::cout << "OldPrinter: " << text << "\n";
    }
};

class IPrinter {
public:
    virtual ~IPrinter() = default;
    virtual void print(const std::string& text) = 0;
};

class PrinterAdapter : public IPrinter {
public:
    explicit PrinterAdapter(OldPrinter& old_printer)
        : old_printer_(old_printer) {}

    void print(const std::string& text) override {
        old_printer_.old_print(text);
    }

private:
    OldPrinter& old_printer_;
};
相关实例：把旧版数据库接口包装成新 DAO 接口；把第三方日志库包装成公司统一日志抽象；把 SDL/平台 API 包成引擎内部统一渲染接口。

装饰器模式

装饰器模式的作用是在不修改原类的前提下，动态给对象附加新职责。它比直接继承更灵活，特别适合“功能叠加组合”的场景。

问题 如果只用继承会怎样 装饰器模式的改进
功能组合越来越多 子类数量爆炸，例如带加密日志、带缓存日志、带加密和缓存日志 通过一层层包装对象来叠加能力，避免类组合爆炸
一个对象可以被多层装饰，每一层都只关注自己新增的职责。
class Stream {
public:
    virtual ~Stream() = default;
    virtual void write(const std::string& text) = 0;
};

class FileStream : public Stream {
public:
    void write(const std::string& text) override {
        std::cout << "write file: " << text << "\n";
    }
};

class StreamDecorator : public Stream {
public:
    explicit StreamDecorator(std::unique_ptr<Stream> stream)
        : stream_(std::move(stream)) {}

protected:
    std::unique_ptr<Stream> stream_;
};

class EncryptStream : public StreamDecorator {
public:
    using StreamDecorator::StreamDecorator;

    void write(const std::string& text) override {
        stream_->write("[encrypt]" + text);
    }
};
装饰器模式和继承最大的区别是：继承是在类型层面扩展，装饰器是在对象层面动态组合。换句话说，它更像“功能拼装”，而不是“新建一个越来越胖的子类”。
相关实例：IO 流增加缓冲、压缩、加密；HTTP 请求处理链附加鉴权、限流、日志；游戏技能效果按“基础攻击 + 暴击 + 火焰附加”逐层包装。

面试回答建议

推荐答题结构

  • 先说这个模式解决什么问题。
  • 再说核心结构和角色关系。
  • 接着给一个真实业务例子。
  • 最后补一句优点、缺点和适用边界。

常见失分点

  • 只会背定义，不会落到场景。
  • 只说优点，不说代价。
  • 把“用了虚函数”误当成“已经设计成模式”。
  • 把模式当成目标，而不是把解决问题当目标。
一个通用收尾表达：我会优先判断系统的变化点在哪里，再决定是否引入某种模式。如果需求很简单，直接写清楚往往更好；只有当模式能明显降低耦合、提升扩展性或测试性时，我才会引入它。
