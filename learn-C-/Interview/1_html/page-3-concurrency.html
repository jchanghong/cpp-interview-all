线程、锁与条件变量

并发问题真正的难点不在 API 名称，而在“什么时候要同步”“同步的粒度多大”“唤醒条件是否正确”。面试官很喜欢用生产者-消费者、线程安全队列、死锁分析来追问这一块。

std::thread

最基础的线程封装。使用时要注意 join/detach 责任，否则析构时会触发终止。

std::jthread

C++20 更现代的线程类型，析构时会自动请求停止并 join，减少忘记回收线程的风险。

条件变量

用于线程间等待和通知，典型用于生产者-消费者模型。

class SafeQueue {
public:
    void push(int value) {
        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_.push(value);
        }
        cv_.notify_one();
    }

    int pop() {
        std::unique_lock<std::mutex> lock(mutex_);
        cv_.wait(lock, [this] { return !queue_.empty(); });
        int value = queue_.front();
        queue_.pop();
        return value;
    }

private:
    std::mutex mutex_;
    std::condition_variable cv_;
    std::queue<int> queue_;
};
wait 配合谓词非常重要，因为条件变量可能发生虚假唤醒。正确思路不是“被唤醒就执行”，而是“被唤醒后重新检查条件是否成立”。
死锁的四个必要条件分别是互斥、占有且等待、不可剥夺、循环等待。实际回答时，可以顺手补上避免方式：固定加锁顺序、缩小临界区、使用 std::lock 一次拿多个锁。

原子、CAS 与 memory_order

这是底层面试很常见的加深追问。真正的重点不是把枚举值背下来，而是明白原子性、可见性和顺序性不是同一件事。

原子操作

适合保护简单共享状态，例如计数器、标志位。它能避免数据竞争，但不自动替代所有复杂同步需求。

CAS

Compare-And-Swap，先比较内存中的当前值是否等于预期值，若相等则更新。很多无锁结构都依赖它。

内存序 直观理解 典型建议
memory_order_relaxed 只保证原子性，不强制顺序与可见性关系 除非真的清楚同步模型，否则不要轻易上来就用
memory_order_acquire 获取语义，确保后续读取看到发布前的结果 常与 release 配对使用
memory_order_release 发布语义，保证之前写入对 acquire 方可见 常用于发布已初始化数据
memory_order_seq_cst 最强一致顺序语义，最容易理解 业务代码里优先保证正确性时非常常见
一个很稳妥的回答方式是：如果没有非常明确的性能瓶颈和同步模型，我会优先用 seq_cst 保证正确性；只有在理解 happens-before 关系且确有收益时，才会考虑 acquire/release 或 relaxed。

C++17 / C++20 / C++23 热点

新标准在面试中的角色通常是“加分项”和“追问项”。如果岗位强调 modern C++，建议至少把 C++20 的核心能力说清楚，再补一句 C++23 的代表性特性。

C++17

optional、variant、any、string_view、结构化绑定。重点追问常在 string_view 的生命周期风险。

C++20

concepts、ranges、coroutines、span、jthread、modules。最值得主动提的是 Concepts 与 Ranges。

C++23

expected、mdspan、print 方向增强，以及 ranges 生态继续完善。最常见的切入点是 expected。

std::expected<int, std::string> parse_positive(std::string_view text) {
    int value = std::stoi(std::string(text));
    if (value <= 0) {
        return std::unexpected("value must be positive");
    }
    return value;
}
expected 和 optional 的关键区别是：optional 只能表达“有值/没值”，而 expected 能显式携带错误信息，因此在工程代码里往往更有表达力。
span 和 string_view 都是不拥有底层数据的视图类型。它们的优势是零拷贝，但风险是底层对象一旦失效，视图就悬空。

工程实践与项目追问

如何避免内存泄漏

优先使用 RAII，把资源交给标准库类型管理。必要时配合 AddressSanitizer、Valgrind、代码审查和单元测试。

如何做性能优化

减少不必要拷贝，善用 move 语义，给容器预留容量，降低锁竞争，减少堆分配，关注缓存局部性。

如何定位线上崩溃

结合 core dump、gdb、日志、最小复现、ASan / TSan，优先缩小问题范围，再定位生命周期、越界或并发竞态。

如何设计线程安全队列

最常见方案是 mutex + condition_variable。若要进一步讨论，可延伸到无锁队列、CAS、ABA 问题与回收机制。

项目追问的一个很实用答题框架是：问题背景 -> 为什么会出错或慢 -> 你做了什么修改 -> 如何验证效果。这个结构比单纯罗列技术名词更有说服力。

面试前 1 小时冲刺清单

必须会讲清楚

  • RAII 与所有权
  • Rule of 3/5/0
  • 左值、右值、move、forward
  • unique_ptr / shared_ptr / weak_ptr
  • vector / unordered_map 的底层原理
  • 虚函数、多态、虚析构
  • 线程、锁、条件变量、原子

建议补一句的加分点

  • string_view / span 是非拥有视图，要注意生命周期
  • Concepts 比 enable_if 更易读
  • Ranges 更适合表达可组合的数据流
  • expected 比 optional 更适合显式返回错误
  • 如果没有明确收益，优先保证并发正确性而不是过早压榨内存序性能
