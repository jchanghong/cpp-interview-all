指针与引用

指针和引用的区别几乎是所有 C++ 面试都会出现的问题，但真正想考的是你是否理解“访问一个对象”和“拥有一个对象”并不是一回事。 指针更像一个独立的地址变量，而引用更像现有对象的一个别名。

维度 指针 引用
是否可为空 可以，常见值是 nullptr 语义上通常不能为空，定义时必须绑定对象
是否可改绑 可以改指向另一个对象 不能改成引用另一个对象
语法使用 需要显式解引用 *p 使用方式与普通对象几乎一样
常见用途 表达可空、可选、动态结构、所有权传递 表达必然存在、避免拷贝、修改实参
指针是“指向对象的一个值”，引用是“对象的别名”。这也是两者语义差异的根源。
void modify_by_pointer(int* p) {
    if (p) {
        *p = 100;
    }
}

void modify_by_reference(int& x) {
    x = 100;
}
面试回答模板：指针和引用都可以间接访问对象，但指针本身是一个可变地址值，能表达“可能为空”与“可改指向”；引用更适合表达“这个对象一定存在，而且这里只是它的别名”。

const、static、inline

这些关键字面试里很爱单拎出来问，因为它们同时涉及类型系统、生命周期和链接属性。 如果回答只停留在“不能修改”或“静态变量在全局区”，通常会显得比较浅。

const int* p

不能通过 p 修改其所指对象，但 p 自己可以指向别处。

int* const p

p 自己不可改指向，但它指向的对象可修改。

const int* const p

既不能改指向，也不能通过它改对象。

成员函数后的 const 是个很常见的追问。它修饰的是隐式的 this 指针类型，相当于承诺“这个函数不会修改对象的逻辑状态”。 
class Counter {
public:
    int value() const { return value_; }   // this 的类型近似为 const Counter* const
    void inc() { ++value_; }

private:
    int value_ = 0;
};

static 的常见场景

  • 修饰局部变量：生命周期延长到程序结束，但作用域仍在函数内部。
  • 修饰全局变量或函数：限制链接范围到当前编译单元。
  • 修饰类成员变量：属于类本身，而不是某个对象。
  • 修饰类成员函数：没有 this 指针，只能访问静态成员。

inline 的现代意义

它早期确实有“建议内联展开”的含义，但现代编译器是否内联主要自己决定。如今更重要的意义是：允许同一个函数在多个翻译单元中定义而不违反 ODR。

封装、继承、多态

这三者看起来像教科书概念，实际上是 C++ 对象模型的入口。面试官通常不只想听定义，而是想听你是否知道“多态成立的条件”和“多态为什么会有运行时开销”。

封装

把实现细节藏起来，只暴露稳定接口。它的价值不是“把成员写成 private”，而是降低模块耦合。

继承

复用基类接口与部分实现，同时建立“is-a”的类型关系。使用不当会导致层次过深、耦合过强。

多态

基类指针或引用指向派生类对象，通过虚函数调用派生实现。多态的本质是动态绑定。

很多候选人会把“重载、重写、隐藏”混在一起。这个点特别容易被顺手追问。
概念 发生位置 核心特征
重载 同一作用域 同名函数参数列表不同，编译期决定调用哪个
重写 继承层次中 派生类覆盖基类虚函数，运行期可能动态绑定
隐藏 继承层次中 派生类同名函数会屏蔽基类同名函数，即使参数不同也可能隐藏

策略模式：把“怎么做”从“谁来做”里拆出来

策略模式是面试里非常常见的行为型设计模式。它解决的核心问题是：同一个业务目标可能有多种实现算法，如果把这些分支都硬编码在一个类里， 往往会导致大量 if-else / switch，后续每新增一种算法都要修改原有类，违反开闭原则。

详细定义：策略模式定义一组可互换的算法，把每个算法分别封装起来，并让它们可以在运行时自由切换。上下文对象只依赖抽象策略接口，不依赖具体算法细节，因此“行为可变”，而“调用方式稳定”。

核心思想

把易变化的算法独立出去，用组合代替把所有逻辑塞进同一个类中。

面试关键词

行为型模式、封装算法族、运行时切换、开闭原则、组合优于继承。

常见考法

“什么时候用策略模式”“和简单工厂有什么区别”“为什么它比 if-else 更好扩展”。

角色 职责 对应到 C++ 中常见写法
Strategy 定义统一算法接口 抽象基类 + 纯虚函数，或可调用对象 / std::function
ConcreteStrategy 实现具体算法 如满减、会员折扣、普通价、不同寻路策略
Context 持有策略对象并在合适时机调用 如订单结算器、导航器、角色 AI 控制器
策略模式把“变化的算法”隔离在具体策略里，让上下文只依赖稳定接口。新增算法时通常新增类，而不是修改旧分支。

适用场景

  • 同一任务存在多套可切换算法，如折扣计算、排序规则、路由选择、角色行为。
  • 原代码里出现大量根据类型或状态分支的 if-else。
  • 希望新增算法时尽量不改已有主流程代码。

优缺点

  • 优点：扩展性好、分支逻辑更清晰、便于单元测试、支持运行时切换。
  • 缺点：类数量会增多，若策略很简单，过度抽象反而会让代码变重。
  • 一个高频追问是：客户端必须知道有哪些策略可选，因此要配合工厂或配置系统统一创建。

一个非常典型的 C++ 例子是“电商结算”。订单总价的计算目标不变，但不同用户或活动对应不同折扣算法。 如果把普通价、会员价、满减价都写进 OrderContext，每新增活动都得改原类；改成策略模式后，结算器只负责调用当前策略。 

#include <iostream>
#include <memory>

class PricingStrategy {
public:
    virtual ~PricingStrategy() = default;
    virtual double calc(double origin_price) const = 0;
};

class NormalPricing : public PricingStrategy {
public:
    double calc(double origin_price) const override {
        return origin_price;
    }
};

class MemberPricing : public PricingStrategy {
public:
    double calc(double origin_price) const override {
        return origin_price * 0.85;
    }
};

class FullReductionPricing : public PricingStrategy {
public:
    double calc(double origin_price) const override {
        return origin_price >= 300.0 ? origin_price - 40.0 : origin_price;
    }
};

class CheckoutContext {
public:
    explicit CheckoutContext(std::unique_ptr<PricingStrategy> strategy)
        : strategy_(std::move(strategy)) {}

    void set_strategy(std::unique_ptr<PricingStrategy> strategy) {
        strategy_ = std::move(strategy);
    }

    double checkout(double origin_price) const {
        return strategy_->calc(origin_price);
    }

private:
    std::unique_ptr<PricingStrategy> strategy_;
};

int main() {
    CheckoutContext ctx(std::make_unique<NormalPricing>());
    std::cout << ctx.checkout(320.0) << "\n";  // 320

    ctx.set_strategy(std::make_unique<MemberPricing>());
    std::cout << ctx.checkout(320.0) << "\n";  // 272

    ctx.set_strategy(std::make_unique<FullReductionPricing>());
    std::cout << ctx.checkout(320.0) << "\n";  // 280
}
这个实现里 CheckoutContext 不知道“会员价怎么算”“满减怎么算”，它只知道自己持有一个 PricingStrategy。 这正是策略模式的价值所在：高层流程稳定，算法细节可替换。

实例 1：游戏 AI

敌人对象可以注入激进、保守、逃跑等不同策略。角色本身不需要知道每种 AI 的细节，只在当前状态下调用行为接口。

实例 2：排序与过滤

日志系统、排行榜、推荐系统里，经常需要根据配置切换排序策略或过滤策略。策略模式能让切换逻辑比硬编码分支更整洁。

面试里一个容易说浅的点是“策略模式就是多态”。更完整的表述应该是：它不仅使用多态，更重要的是把一组可替换算法从上下文中解耦出来，让行为变化被封装和组合。
面试回答模板：策略模式属于行为型模式，它把一组算法封装成独立策略类，并通过统一接口对外提供能力。上下文依赖抽象策略而不是具体实现，因此可以在运行时切换算法。典型场景包括折扣计算、支付方式、路径规划、角色 AI 等。它的优点是消除复杂分支、符合开闭原则、便于测试；缺点是类会变多，简单场景下可能显得过度设计。

虚函数、虚表与析构

这是 C++ 面试里最经典、也最容易区分深浅的一类问题。真正重要的不只是“类里有个 vptr”，而是理解为什么基类析构要虚、为什么构造/析构中不建议调用虚函数，以及多态到底依赖什么条件。

通过基类指针或引用调用虚函数时，程序会先经由对象中的虚表指针找到对应虚表，再定位到最终函数地址。
class Base {
public:
    virtual ~Base() = default;
    virtual void speak() const {
        std::cout << "Base\n";
    }
};

class Derived : public Base {
public:
    void speak() const override {
        std::cout << "Derived\n";
    }
};
基类析构函数通常应设为虚函数，因为如果你通过 Base* 删除一个 Derived 对象，而析构不是虚的，就只会执行基类析构，派生类资源可能泄漏。
构造和析构期间不要依赖虚函数实现多态。构造时派生部分还没构造完，析构时派生部分已经销毁，多态语义并不完整。

拷贝控制、RAII 与 Rule of 3/5/0

拷贝控制是 C++ 与很多语言差异很大的地方。只要一个类自己管理资源，就不能只写一半。很多面试官会通过这个点判断候选人是否真的理解对象生命周期。

Rule of 3

如果类需要自定义析构、拷贝构造、拷贝赋值中的一个，通常另外两个也要认真定义。

Rule of 5

在 C++11 之后，还要一并考虑移动构造和移动赋值，因为资源的“转移”也是对象语义的一部分。

Rule of 0

最佳实践通常是根本不自己管理资源，而是交给 std::string、std::vector、std::unique_ptr 之类的 RAII 类型。

class Buffer {
public:
    explicit Buffer(std::size_t size)
        : size_(size), data_(new int[size]{}) {}

    ~Buffer() {
        delete[] data_;
    }

    Buffer(const Buffer& other)
        : size_(other.size_), data_(new int[other.size_]) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    Buffer& operator=(const Buffer& other) {
        if (this == &other) {
            return *this;
        }
        int* new_data = new int[other.size_];
        std::copy(other.data_, other.data_ + other.size_, new_data);
        delete[] data_;
        data_ = new_data;
        size_ = other.size_;
        return *this;
    }

private:
    std::size_t size_;
    int* data_;
};
面试里更推荐补一句：如果这是实际工程代码，我更倾向于直接用 std::vector<int> 或 std::unique_ptr<int[]>，这样就更接近 Rule of 0，维护成本也更低。

new/delete、malloc/free 与内存布局

这部分很常作为快问快答出现。定义本身不难，但如果你能自然提到“构造/析构”和“对象语义”，答案会明显更完整。

new / delete

这是 C++ 的运算符。new 先分配原始内存，再调用构造函数；delete 先调用析构函数，再释放内存。

malloc / free

这是 C 标准库函数，只处理原始内存，不会自动构造或析构对象，因此不适合直接承担现代 C++ 的对象管理职责。

面试中说“内存分区”时，不同平台的具体实现细节会有差异，但大致模型可以这样表达。
