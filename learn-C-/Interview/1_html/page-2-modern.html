右值引用、move 语义与完美转发

move 语义几乎是现代 C++ 面试的标志性话题。这里真正考查的不是会不会背“左值右值”，而是你能否解释“为什么移动能更高效”，以及“什么情况下不能盲目移动”。

左值

通常有名字、可取地址、生命周期较稳定，例如局部变量、全局变量、容器中的元素。

右值

临时对象或将亡值，往往适合作为资源转移的来源。

右值引用

T&& 主要用于绑定右值，从而支持移动语义和完美转发。

class StringBuffer {
public:
    explicit StringBuffer(std::size_t n)
        : size_(n), data_(new char[n]{}) {}

    StringBuffer(StringBuffer&& other) noexcept
        : size_(other.size_), data_(other.data_) {
        other.size_ = 0;
        other.data_ = nullptr;
    }

private:
    std::size_t size_ = 0;
    char* data_ = nullptr;
};
std::move 本身不会移动任何东西，它只是把一个表达式转换成右值表达式。真正是否发生移动，要看目标类型是否提供了合适的移动构造或移动赋值。
被 move 之后的对象仍然必须保持“有效但未指定”的状态。也就是说它必须还能析构、还能重新赋值，但你不能假定它一定为空。 
template <typename T, typename... Args>
std::unique_ptr<T> make_owned(Args&&... args) {
    return std::unique_ptr<T>(new T(std::forward<Args>(args)...));
}
面试回答模板：move 语义的本质是把资源所有权从源对象转移到目标对象，避免昂贵的深拷贝。右值引用是它的语言基础，而完美转发则是在模板中保留参数原有值类别的机制。

智能指针

智能指针的重点不是把裸指针换个壳，而是把“所有权语义”显式化。现在很多面试官会特别注意你是否能正确说出“谁拥有对象”“谁只是观察对象”。

类型 所有权 典型用途 注意点
unique_ptr 独占 默认首选，清晰表达唯一拥有关系 不可拷贝，只能移动
shared_ptr 共享 多个所有者共同决定对象生命周期 有计数开销，容易循环引用
weak_ptr 不拥有 观察 shared_ptr 管理的对象，打破循环 使用前要 lock() 
struct Node {
    int value = 0;
    std::shared_ptr<Node> next;
    std::weak_ptr<Node> prev;  // 防止双向链表形成循环引用
};
大多数情况下应该优先使用 unique_ptr。如果你一开始就使用 shared_ptr，往往意味着所有权设计还没有想清楚。

STL 容器：为什么面试总爱问 vector 和 unordered_map

因为容器问题能同时考你对复杂度、底层结构、缓存局部性和工程取舍的理解。真正好的答案不只是背 O(1) 或 O(log n)，而是要说明“为什么在真实代码里会这样选”。

vector

  • 连续内存，随机访问快，缓存局部性好。
  • 尾部插入删除通常快，中间插入删除需要搬移元素。
  • 扩容时可能整体搬迁，导致迭代器、指针、引用失效。

list

  • 双向链表，任意位置插删较快。
  • 不支持随机访问，节点分散导致缓存局部性差。
  • 虽然理论插删复杂度好看，但实际工程里不一定比 vector 快。

map

  • 通常基于红黑树，键有序。
  • 查找、插入、删除一般是 O(log n)。
  • 适合有序遍历、范围查询。

unordered_map

  • 通常基于哈希表，平均查找 O(1)。
  • 无序，最坏情况可能退化到 O(n)。
  • rehash 会带来元素搬移与迭代器失效问题。
一个很好的补充表达是：很多场景下 vector 比 list 更快，不是因为复杂度更低，而是因为连续内存更符合 CPU cache 的访问模式。

迭代器失效

这是一个非常典型的“看似小题，实际很工程”的问题。因为它考查你是否真的写过容器代码，尤其是边遍历边删除、插入扩容、rehash 这些真实场景。

容器 常见失效场景 面试高频提醒
vector 扩容、插入、删除后大范围失效 预留容量 reserve 可以减少扩容带来的失效和拷贝
list 通常只有被删除节点失效 其他节点迭代器通常仍有效
unordered_map rehash 后迭代器可能失效 负载因子变化可能触发 rehash 
for (auto it = nums.begin(); it != nums.end(); ) {
    if (*it % 2 == 0) {
        it = nums.erase(it);  // 不能简单写 ++it 后再 erase
    } else {
        ++it;
    }
}

模板、SFINAE 与 Concepts

这部分是很多中高级岗位非常关注的区域。模板不只是“泛型写法”，它本质上是编译期代码生成和接口约束机制。过去大家常用 enable_if 和 SFINAE，C++20 以后则更推荐用 Concepts。

SFINAE

替换失败不是错误。模板参数替换时，如果某个候选不成立，编译器会继续找其他候选，而不是立刻报错。

Concepts

把模板约束显式写出来，让接口意图更清晰，报错信息也更友好。这是现代 C++ 面试里非常值得主动提到的升级点。

template <typename T>
concept Addable = requires(T a, T b) {
    a + b;
};

template <Addable T>
T add_twice(const T& a, const T& b) {
    return a + b + b;
}
面试里可以这样表述：SFINAE 是传统模板约束手段，Concepts 是 C++20 带来的更现代、更可读、更接近“接口契约”的方式。

auto、decltype、constexpr、lambda

auto

用于自动类型推导，能减少冗长类型声明，但也要避免把真实语义藏得太深。

decltype

推导表达式类型，常用于泛型编程和复杂返回值场景。

constexpr

强调编译期可求值能力。现代 C++ 中它的能力越来越强，已经不只是“常量”这么简单。

lambda

本质是匿名函数对象。面试追问重点往往在捕获方式、生命周期和异步场景下的风险。

int base = 10;

auto plus_by_value = [base](int x) {
    return x + base;
};

auto plus_by_ref = [&base](int x) {
    base += 1;
    return x + base;
};
在线程池、异步回调、协程等场景中，引用捕获要特别谨慎。如果外部对象先销毁，lambda 内部就可能出现悬空引用。
